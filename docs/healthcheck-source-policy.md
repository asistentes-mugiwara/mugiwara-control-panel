# Healthcheck source policy

## Purpose
Phase 15.2c closed the final foundation slice before live Healthcheck source adapters. It defines the source-policy contract that adapters must satisfy. Phase 15.3a started live reads with the `vault-sync` adapter; Phase 15.3b added the `backup-health` adapter; Phase 15.4a added the `project-health` local repo adapter; Phase 15.4b adds the Zoro-owned producer for that fixed project-health manifest. Issue #43 adds the user-level `mugiwara-project-health-status.timer` and `scripts/install-project-health-status-user-timer.sh` to refresh that manifest periodically. Phase 15.5a adds the fixed Franky-owned `gateway-status` manifest reader for global and per-Mugiwara gateway summaries. Phase 15.5b adds the Franky-owned gateway status producer and user-level `mugiwara-gateway-status.timer`, installed by `scripts/install-gateway-status-user-timer.sh`, to refresh that manifest periodically. Phase 15.6a adds the fixed Franky-owned `cronjobs-status` manifest reader for the global cronjobs source; it consumes only manifest timestamp/status plus per-job `last_run_at`, `last_status` and `criticality` semantics and still does not inspect prompts, commands, chat IDs, logs or outputs. Phase 15.6b adds the Franky-owned cronjobs status producer and user-level `mugiwara-cronjobs-status.timer`, installed by `scripts/install-cronjobs-status-user-timer.sh`, to refresh that manifest periodically from allowlisted Hermes profile cron registries. It serializes only aggregate status/timestamp plus per-job `last_run_at`, `last_status` and `criticality`, and does not serialize job names, owner profiles, prompt bodies, commands, delivery targets, chat IDs, logs, stdout/stderr, raw outputs, host paths, tokens or credentials. Phase 15.7a adds a fixed 1 MiB pre-parse size limit per allowlisted `jobs.json`; oversized registries fail closed before `json.loads` and their content is not logged or serialized. These adapters consume fixed reviewed manifests and route through the existing registry sanitizer.

## No generic host console
Healthcheck must not become a generic host console. New source code in `apps/api/src/modules/healthcheck` must not introduce generic shell, process, command execution, arbitrary URL-fetch adapters or generic filesystem discovery without a reviewed, allowlisted phase.

Blocked by `npm run verify:healthcheck-source-policy`:
- generic subprocess or shell usage;
- `exec()` / `eval()` style execution;
- generic `requests`, `httpx` or `urllib.request` URL fetches inside the Healthcheck module;
- command-parameter based host adapters;
- generic filesystem discovery or reads such as `glob`, `os.listdir`, `os.scandir`, `os.walk`, ambient `Path.home()` / `Path.cwd()`, recursive `rglob()` or direct `open()` in the Healthcheck module.

Adapters remain explicit, source-family-specific and reviewed. No live manifest reads are implemented in Phase 15.2c. Phase 15.3a allows the fixed `vault-sync` manifest reader in `source_adapters.py`; Phase 15.3b also allows the fixed `backup-health` manifest reader; Phase 15.4a also allows the fixed `project-health` repo-local manifest reader. Phase 15.4b adds `scripts/write-project-health-status.py` as the reviewed producer for that fixed manifest; it may query Git outside the backend, but writes only status/result, updated_at and boolean workspace/main/sync semantics. Issue #43 installs it through the user-level `mugiwara-project-health-status.timer`; the service runs `npm run write:project-health-status` from the repo root, does not run `git fetch`, does not pass `--repo`/`--output`, and `remote_synced` compares `HEAD` with the local upstream ref. If live network refresh is needed later, it must be a separate reviewed runner/follow-up. Phase 15.5a also allows the fixed `gateway-status` manifest reader for `hermes-gateways` and allowlisted `gateway.<mugiwara-slug>` sources; it consumes only `updated_at` and boolean/enum active state, and still does not execute systemd from the backend. Phase 15.5b adds `scripts/write-gateway-status.py` as the reviewed producer for that fixed gateway manifest; it runs outside the backend, only checks allowlisted `hermes-gateway-<slug>.service` active state via `systemctl --user is-active`, writes only status/result, updated_at and per-slug `active` booleans, fsyncs the parent directory after `os.replace`, and is installed by `mugiwara-gateway-status.timer` through `scripts/install-gateway-status-user-timer.sh`. The CLI `--output` option is retained only for tests/manual controlled runs; the installed service uses the fixed package script, passes no alternate output path and caps abnormal waits with `TimeoutStartSec=30s`. It does not inspect journal output, unit file contents, PIDs, command lines, env values, logs, stdout/stderr or alternate output paths. Phase 15.6a also allows the fixed `cronjobs-status` manifest reader for the global `cronjobs` source; it consumes only `updated_at`, manifest result and per-job safe status/freshness/criticality semantics, degrades empty/missing/unreadable registries visibly, and does not expose job names, owner profiles, prompt bodies, commands, delivery targets, chat IDs, logs, stdout/stderr or raw outputs. Phase 15.6b adds `scripts/write-cronjobs-status.py` as the reviewed producer for that fixed cronjobs manifest. It runs outside the backend, reads only allowlisted Hermes profile cron registries, writes only status/result, updated_at and per-job last-run/status/criticality semantics, and is installed by `mugiwara-cronjobs-status.timer` through `scripts/install-cronjobs-status-user-timer.sh`. It does not use `cronjob list`, does not inspect prompt bodies beyond parsing the registry file, and does not serialize job names, owner profiles, prompts, commands, delivery targets, chat IDs, logs, stdout/stderr, raw outputs, host paths, tokens or credentials. These phases still do not add GitHub issue/PR counts or last-verify aggregation.

## Text field sanitization
Future adapters must sanitize summaries before handing them to Healthcheck. As defense in depth, `HealthcheckSourceRegistry` ignores adapter-provided labels and always resolves `label` from backend-owned `HEALTHCHECK_SOURCE_LABELS[source_id]`. It also applies a final sensitive-marker filter to allowed textual fields: `summary`, `warning_text`, `source_label` and `freshness_label`.

If those fields contain host paths, `.env`, tokens, credentials, cookies, raw output markers, stdout/stderr, commands, tracebacks, journals, prompts, chat IDs, delivery targets, Git diffs, untracked files or internal remotes, the registry replaces the value with a generic safe fallback. This preserves status/severity/freshness semantics while preventing sensitive textual content from reaching the public read model.

## Manifest ownership and safe location class
Future live adapters may consume only sanitized summaries from reviewed sources. They must never expose raw host details.

Phase 15.8 closes the block canon through Phase 15.7b: fixed readers exist for all five planned source families, and periodic producers/runners exist for `project-health`, `gateway-status` and `cronjobs-status`. `vault-sync-status` and `backup-health-status` producers remain separate operational follow-ups; until then their readers must degrade absent/unreadable manifests visibly instead of guessing healthy state. Phase 18.0 reconciles those follow-ups into a dedicated producer block: producer and runner/timer stay separated per source, `vault-sync` goes before `backup-health`, and each producer/runner PR needs Franky + Chopper review before merge.

Source manifests or wrappers do not include `stdout`, `stderr`, `raw_output`, `command`, `traceback`, `pid`, `unit_content`, `journal`, absolute host paths, `backup_path`, `included_path`, `prompt_body`, `chat_id`, delivery targets, tokens, cookies, credentials, `.env`, Git diffs, untracked file lists or internal remotes.

## Initial freshness thresholds
Thresholds are intentionally backend-owned policy before Phase 15.3+ live adapters start mapping timestamps to `pass`/`warn`/`fail`.

- `vault-sync`: warn after 90 minutes, fail after 360 minutes.
- `project-health`: warn after 120 minutes, fail after 480 minutes.
- `backup-health`: warn after 1800 minutes, fail after 4320 minutes.
- `hermes-gateways` and `gateway.<mugiwara-slug>`: warn after 15 minutes, fail after 60 minutes.
- `cronjobs`: warn after 180 minutes, fail after 720 minutes.

These thresholds are not applied to live data in Phase 15.2c. Phase 15.3a/15.3b apply them only to fixed `vault-sync` and `backup-health` manifests. Phase 15.4a applies the `project-health` thresholds only to the fixed repo-local status manifest; Phase 15.4b only refreshes that manifest timestamp and safe boolean semantics. Phase 15.5a applies gateway thresholds only to the fixed gateway status manifest; stale or absent data degrades visibly and per-gateway state is derived only from allowlisted slugs. Phase 15.5b refreshes the gateway manifest timestamp and per-gateway `active` booleans through `mugiwara-gateway-status.timer`. Phase 15.6a applies `cronjobs` thresholds only to the fixed cronjobs status manifest and critical job `last_run_at` values; stale, absent, empty or unreadable cron registries degrade visibly and never become healthy output. Phase 15.6b refreshes the cronjobs manifest through `mugiwara-cronjobs-status.timer`; Phase 15.7a makes oversized cron registries fail closed before JSON parsing; the frequent `vault-sync` cron maps to `critical`, longer-cadence recurring jobs map to `normal`, and one-shot future jobs and normal jobs without a first recorded run are excluded from the recurring health registry. Future adapters must parse timestamps explicitly, normalize to the existing Healthcheck freshness vocabulary and degrade absent/unreadable data to `not_configured`, `unknown` or `stale`, never to healthy output.

## Verify
Run after changes that touch Healthcheck source adapters, docs or source-policy boundaries:

```bash
npm run verify:healthcheck-source-policy
```

For implementation phases that also touch the API read model, run the backend test suite and existing perimeter guardrails as well.

## Phase 18.1/18.2 vault-sync status producer and runner
Phase 18.1 adds `scripts/write-vault-sync-status.py` and `npm run write:vault-sync-status` as the reviewed producer for `/srv/crew-core/runtime/healthcheck/vault-sync-status.json`; it intentionally had no unit/timer in Phase 18.1. The producer runs the fixed Franky-owned `/srv/crew-core/scripts/vault-sync.sh` operational source outside the backend and consumes only the exit code. It writes only `status`, `result`, `updated_at` and, on explicit success, `last_success_at`; it does not serialize stdout, stderr, raw output, logs, paths, branch names, remotes, SHAs, diffs, tokens, credentials or `.env` values.

The producer writes atomically with a temp file in the same directory, file fsync, `os.replace`, file mode `0640`, directory mode `0750` and parent-directory fsync. `npm run verify:vault-sync-status-producer` guards this contract. Phase 18.2 adds the user-level `mugiwara-vault-sync-status.timer`, installed by `scripts/install-vault-sync-status-user-timer.sh`; it runs `npm run write:vault-sync-status` from the fixed repo root every 20 minutes, uses `TimeoutStartSec=620s` aligned with the producer internal 600s timeout, and does not pass `--output`, `--sync-script` or `--timeout-seconds`. The runner must not serialize stdout/stderr, raw output, logs, paths, branches, remotes, tokens, credentials or `.env` values.

## Phase 18.3 backup-health status producer
Phase 18.3 adds `scripts/write-backup-health-status.py` and `npm run write:backup-health-status` as the reviewed producer for `/srv/crew-core/runtime/healthcheck/backup-health-status.json`; it intentionally has no unit/timer in Phase 18.3. The producer does not run backups: it observes only the fixed local backup artifact directory, validates the latest checksum with `sha256sum -c` while discarding stdout/stderr, and serializes only `status`, `result`, `updated_at`, optional `last_success_at`, `checksum_present` and `retention_count`.

The producer writes atomically with a temp file in the same directory, file fsync, `os.replace`, file mode `0640`, directory mode `0750` and parent-directory fsync. `npm run verify:backup-health-status-producer` guards this contract. Success is fail-closed: `status=success` requires a valid latest checksum and `retention_count >= 4`; absence, empty source, checksum absence/failure or insufficient retention degrades to `failed`/`warning` and omits `last_success_at`. It does not serialize archive names, paths, sizes, hashes, Drive targets, stdout/stderr, logs or raw output.

## Phase 18.4 backup-health status runner
Phase 18.4 adds the user-level `mugiwara-backup-health-status.timer`, installed by `scripts/install-backup-health-status-user-timer.sh`, to refresh `/srv/crew-core/runtime/healthcheck/backup-health-status.json` periodically. The runner runs `npm run write:backup-health-status` from the fixed repo root every 8 hours, uses `TimeoutStartSec=120s`, and does not pass `--output` or `--backups-dir` to the producer.

The runner does not run backups, `/srv/crew-core/scripts/system-backup.sh`, archive tooling, compression tooling, Drive upload tooling or `rclone`; it only refreshes the safe aggregate manifest. It also does not serialize archive names, backup paths, included paths, hashes, sizes, stdout/stderr, logs, raw output, tokens, credentials or `.env` values. `npm run verify:backup-health-status-runner` guards the unit, timer, installer, docs and Phase 18.3/18.4 boundary.

## Phase 18.5 Healthcheck producers closeout
Phase 18.5 closes the Healthcheck producers block canonically. The previously pending `vault-sync-status` and `backup-health-status` follow-ups are now producer+runner complete: `mugiwara-vault-sync-status.timer` refreshes `vault-sync-status.json` every 20 minutes and `mugiwara-backup-health-status.timer` refreshes `backup-health-status.json` every 8 hours. All five Healthcheck manifest families now have fixed readers and active producer/timer coverage outside the backend.

The canonical boundary remains unchanged: Healthcheck backend code consumes only fixed safe manifests and still does not execute shell, Git, systemd, Docker, backup jobs or filesystem discovery. Future Healthcheck sources must start from a new explicit plan with owner, manifest shape, freshness thresholds, guardrails, leakage tests and Franky + Chopper review.

## Phase 18.6 Honcho/Docker operational producers
Phase 18.6 adds `scripts/write-docker-runtime-status.py` and `scripts/write-honcho-status.py` as reviewed producers for issue #129's remaining live operational gap. The Docker producer writes `/srv/crew-core/runtime/healthcheck/docker-runtime-status.json` from a fixed allowlist of critical containers (`honcho-api`, `honcho-database`, `honcho-redis`) and serializes only `status`, `result`, `updated_at` and per-container `running`/`health` values. It does not inspect Docker logs, `docker inspect`, mounts, labels, env, networks, PIDs, command lines, ports, IDs, stdout/stderr or raw output.

The Honcho producer writes `/srv/crew-core/runtime/healthcheck/honcho-status.json` from the sanitized Docker runtime manifest plus the fixed local `/health` endpoint. It serializes only API/DB/Redis `ok` booleans plus timestamp/result. It does not read Honcho `.env`, credentials, connection strings, compose env files, database URLs, Redis URLs, endpoint payload details, Docker internals, logs, stdout/stderr or raw output. `npm run verify:docker-runtime-status-producer` and `npm run verify:honcho-status-producer` guard this contract.

The user-level `mugiwara-honcho-docker-status.timer`, installed by `scripts/install-honcho-docker-status-user-timer.sh`, refreshes both manifests every 2 minutes by running `npm run write:docker-runtime-status` before `npm run write:honcho-status` from the fixed repo root. The runner does not pass `--output`, `--docker-runtime-manifest` or alternate paths, and `npm run verify:honcho-docker-status-runner` guards the installed surface.
